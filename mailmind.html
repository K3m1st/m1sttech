MailMind

What happens when you take a Berkeley homework assignment and point it at your own inbox

// TL;DR

MailMind is a proof of concept, not a product. It started as a Berkeley ML homework assignment and became a phishing classifier trained on my own Gmail inbox — a darkweb-exposed address that's been receiving the same campaigns for years. V1 used Naive Bayes, hand-labeled 585 emails, and ran on an HP Mini PC auto-quarantining phishing. V2 rebuilt with a 96K-parameter character-level CNN, expanded to 840 emails, and hit 0.994 ROC-AUC. The biggest lesson: data quality beats model architecture. Fixing 20 mislabeled emails improved performance more than adding 100 new ones. Collecting real phishing from Gmail's spam folder mattered more than switching from NB to CNN.

01 // The Story

From Homework to Homelab

My Gmail address has been on the darkweb for years. Same phishing campaigns, same credential theft attempts, same CarShield and Fidelity Life scams cycling through on repeat. When a Berkeley ML homework asked me to build a Naive Bayes text classifier, I thought: why train on a textbook dataset when I have a real one sitting in my inbox?

1
Berkeley ML Coursework

The Spark

Implemented a Naive Bayes classifier on the 20 Newsgroups dataset for Berkeley MICS. Text vectorization, conditional probability, evaluation metrics. Standard homework — but the idea wouldn't leave my head: what if I pointed this at real email?

2
MailMind V1 — The 3-Day Sprint

Naive Bayes on Real Gmail

Built the entire pipeline in 72 hours — IMAP ingestion, preprocessing, classifier, Flask dashboard, auto-quarantine poller, Docker deployment. Ported the homework code directly into production. First attempt used 17,536 synthetic emails from HuggingFace. Looked amazing in testing: 0.97 F1. Deployed against real Gmail: 59% of legitimate emails flagged as phishing. Synthetic data poisoned the model. Pivoted to pure Gmail data — 64 hand-labeled emails outperformed 17,536 synthetic ones. Scaled to 585 labels with active learning.

3
MailMind V2 — Going Deeper

Character-Level CNN

NB plateau'd at 81% phishing recall. Rebuilt with a 96K-parameter character-level CNN with multi-scale convolutions. Expanded dataset to 840 emails by auditing Gmail's spam folder — pulled 263 emails, manually reviewed each one, removed 8 false positives, kept 255 genuine phishing. No synthetic data. Real data, carefully labeled.

07 // What This Actually Is

A Proof of Concept, Not a Product

Let me be honest about what MailMind is and isn't. My inbox is predictable — the same phishing campaigns have been hitting it for years. CarShield, Fidelity Life, Endurance Warranty, credential theft templates that barely change between runs. Many of the Gmail-collected phishing emails come from the same spam campaigns, which means near-duplicate emails likely appear across train and test folds. The 0.994 ROC-AUC should be interpreted with that caveat.

At 250 hand-labeled emails, model performance dropped. Not because the model got worse — because my labels got worse. Twenty emails were mislabeled: aggressive marketing (PlayStation deals, gaming promotions) marked as phishing. Same senders had opposite labels. Fixing 20 labels improved F1 more than adding 100 new ones. That's the real lesson of this project. Not the CNN architecture. Not the augmentation pipeline. Understanding your data matters more than tuning hyperparameters.

V2 has never been deployed as a live filter. V1 ran on an HP Mini PC auto-quarantining emails — V2 exists as a research artifact. The boundary between "phishing" and "aggressive spam" is subjective. 840 emails is small by ML standards. Performance on novel phishing tactics is unknown. This project taught me more about the practice of machine learning than any course — but it's a proof of concept built from homework, not a production security tool.
