Homelab.

Defense in depth — from DNS filtering to AI-powered alert triage

// 01

Six Layers Deep

Build like someone's already in. The perimeter is solid — VPN-only access, DNS filtering, encrypted upstream DNS. What was missing: visibility and reasoning.

01

DNS Filtering

Pi-Hole blocks known malicious and ad domains at the network level.

02

DNS Privacy

DNS-over-HTTPS prevents ISP and upstream DNS snooping.

03

Remote Access

Pi-VPN provides encrypted tunnel access. No services exposed to the internet.

04

Network IDS

05

Host Monitoring

Wazuh agents on every device (Ubuntu, macOS, Raspberry Pi). File integrity, rootkit detection, CIS benchmarks, vulnerability scanning.

06

SIEM/XDR

Wazuh Manager correlates host and network events into a single pane of glass. Suricata feeds into Wazuh for unified alerting.

// 02

Thousands of Alerts, One Pair of Eyes

A typical day generates ~7,800 alerts across all severity levels. Level 3 Suricata protocol decodes, generic network noise, routine system events. The signal is buried in the noise. Manual triage is unsustainable for a homelab. But you can't fully automate response without confidence. The answer: an AI layer that enriches, reasons, and flags — a junior analyst that works 24/7 and escalates the edge cases.

Alert Poller

Polls alerts.json every 10 seconds. Filters at level 5+ — cutting ~4,800 daily low-level noise alerts. Skips internal-only traffic that doesn't need reputation lookups.

Enrichment

Three data sources layered onto each alert:

  • GeoIP (ip-api.com) — country, ISP, ASN for external IPs
  • AbuseIPDB — abuse confidence score, total reports, usage type
  • Context Window — surrounding alerts from the same agent within ±5 minutes (capped at 10 for the AI prompt)

Plus a dynamic agent registry that refreshes from the Wazuh API every 5 minutes — new devices appear automatically.

AI Analysis

Claude Haiku (chosen for cost efficiency at high alert volume) receives the enriched alert and returns structured JSON: summary, risk level (Low/Medium/High/Critical), risk reasoning, MITRE ATT&CK technique mapping, and a specific recommended action. Prompted as a senior SOC analyst reviewing a homelab — "avoid over-alarming on routine system noise."

// 04

Why It Works This Way

Haiku over Sonnet

Alert volume can spike to thousands per day. Haiku is fast and cost-efficient for a continuous triage loop. Structured prompts ensure consistent output regardless of model tier.

Poll alerts.json, not OpenSearch

Reading directly from the Wazuh manager container is simpler and more reliable than authenticating to OpenSearch. Same data, fewer dependencies.

Suricata native, Wazuh in Docker

Suricata needs raw packet access to the NIC via AF_PACKET. Docker would add an abstraction layer on the critical packet capture path. Wazuh's Java/OpenSearch/Node.js stack benefits from container isolation and easy version upgrades.

Enrich first, reason second

Raw alerts are noise. GeoIP + reputation + temporal context transforms a "port scan detected" into "port scan from a Russian hosting provider with 87% abuse score, preceded by 3 SSH failures in the last 5 minutes." The AI layer reasons over signal, not noise.

// 05

Where It's Going

Phase 4 (Discord notifications) is working. Phase 5 is auto-response — iptables blocking on Critical severity with confidence threshold guardrails. The philosophy: automate enrichment and analysis completely, but keep a human in the loop for response until confidence is proven. Ship fast, iterate, add complexity only after the system works.
